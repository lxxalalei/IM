# 项目工作规则

版本：v0.1  
日期：2026-05-10  
适用范围：本 Flutter 飞书类 IM 项目的规划、设计、编码、测试、交付全过程

## 1. 置信度审查机制

在推进本项目的任何关键策略、架构决策、实现方案、里程碑计划或交付结论前，必须执行以下审查循环：

> Are you 100% confident in this strategy? If not, find all possible loopholes, suggest proper fixes and run this loop until you are factually 100% confident in the new strategy.

## 2. 执行解释

由于软件工程中无法保证绝对意义上的全知和零风险，本项目将该规则落地为以下可执行机制：

1. 每次提出或执行关键方案前，先进行置信度自检。
2. 如果存在不确定点，必须列出已知漏洞、边界条件、风险和假设。
3. 针对每个漏洞提出修正方案。
4. 用修正后的方案重新审查。
5. 只有当“没有已知重大漏洞、剩余风险可接受且有验证路径”时，才继续推进。

## 3. 每次开发前的检查清单

- 当前目标是否清晰？
- 是否和 PRD、已有代码结构、用户最新要求一致？
- 是否存在架构、状态管理、UI、平台兼容、性能、测试或后续扩展风险？
- 是否会引入不必要的复杂度？
- 是否有更贴近 Flutter 和项目现状的实现方式？
- 是否有明确验收标准？
- 是否需要先验证环境或运行测试？

## 4. 每次交付前的检查清单

- 改动是否完成了用户真正要求的目标？
- 是否遗漏了边界状态、空状态、错误状态或响应式表现？
- 是否运行了可行的验证命令？
- 如果无法验证，是否明确说明原因？
- 是否避免了破坏用户已有改动？
- 是否留下了后续可执行的下一步？

## 5. 项目约定

后续执行本项目时，默认遵守本规则。  
如果新决策与本规则冲突，必须优先进行风险说明，并获得用户明确确认后再调整。
